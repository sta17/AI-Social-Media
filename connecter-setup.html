Server config HTTPS — encrypt traffic ▼

Without HTTPS, JWT tokens and passwords cross the network in plaintext. This doesn't matter on localhost, but is critical for any network-accessible deployment.

Option A — Caddy reverse proxy (recommended)

# Caddyfile
your-domain.com {
    reverse_proxy localhost:8090
}

Caddy automatically provisions and renews Let's Encrypt certificates. Run caddy run alongside PocketBase.

Option B — PocketBase built-in TLS

./pocketbase serve --https

Works for simple setups but requires the server to be reachable on port 443 and have a valid domain.

After enabling HTTPS — update the PB_URL constant at the top of connecter-social-media.html from http:// to https://.

⚠ If you're running on a local network only (home server, LAN), HTTPS is less urgent but still good practice. Self-signed certificates work with some browser configuration.

PocketBase admin Configure SMTP for password reset & verification emails ▼

Password reset and email verification both require PocketBase to be able to send email. Without SMTP configured, these flows silently fail. The forum now shows a persistent banner on login when an account is unverified — but the banner's "Resend email" button only works if SMTP is set up.

Steps in PocketBase admin UI (/_/):

  • Go to Settings → Mail settings
  • Enable Use SMTP mail server
  • Fill in your SMTP host, port, username, and password
  • Set a Sender name and Sender address (e.g. Connecter <[email protected]>)
  • Click Save changes, then use Send test email to verify

Free SMTP options: Resend, Brevo (Sendinblue), Mailgun free tier, or your Gmail account with an App Password.

⚠ Gmail App Passwords: Go to Google Account → Security → 2-Step Verification → App passwords. Use smtp.gmail.com port 587 with TLS.

PocketBase admin Token lifetime & session settings ▼

By default PocketBase JWTs are valid for 30 days. For a private forum this is probably fine, but you can tighten it.

In PocketBase admin UI (/_/):

  • Go to Collections → users → Options (gear icon)
  • Set Auth token duration — e.g. 604800 for 7 days, 86400 for 1 day
  • Set Min password length — recommend at least 10 for a production forum

Shorter token lifetimes mean users get logged out more often. The forum's auth-refresh call on page load extends the session automatically as long as the old token hasn't fully expired yet.

⚠ If you shorten the token lifetime significantly (e.g. to 1 day), make sure the token expiry handling code change above is implemented first, otherwise expired sessions will cause silent breakage.
